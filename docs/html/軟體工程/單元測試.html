單元測試

確保軟體品質的關鍵活動

測試的價值

  • 發現缺陷：在軟體上線前發現錯誤
  • 防止回歸：確保新變更不會破壞現有功能
  • 文件化行為：測試案例作為軟體行為的文件
  • 重構信心：有充分測試覆蓋才能安全重構
  • 持續交付基礎：自動化測試是 CI/CD 的基石

測試金字塔

測試金字塔是指導測試策略的重要原則：

  • 底部（單元測試）：數量多、執行快、成本低，測試最小可測試單元（函數或類別）
  • 中部（整合測試）：數量適中、驗證組件協作
  • 頂部（E2E測試）：數量少、執行慢、接近真實使用

單元測試特性

  • 隔離性：每個測試獨立運行，不依賴其他測試
  • 快速：執行時間通常在毫秒級
  • 確定性：相同的輸入總是產生相同的輸出
  • 可維護性：測試代碼應該與被測試代碼一樣整潔

測試驅動開發（TDD）

TDD 的循環：紅（寫一個失敗的測試）→ 綠（寫最簡單的程式碼讓測試通過）→ 重構（改善程式碼品質）。這個循環確保了測試覆蓋率，也驅動了更好的設計。

測試範例

# 測試對象：簡單的計算器函數 def add(a, b): return a + b def subtract(a, b): return a - b # 測試案例 def test_add(): assert add(2, 3) == 5 assert add(-1, 1) == 0 def test_subtract(): assert subtract(5, 3) == 2
